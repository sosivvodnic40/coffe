## ВВЕДЕНИЕ

Сфера общественного питания в Республике Казахстан активно развивается. Заведения формата coffee shop и ресторанов с авторской кухней конкурируют не только качеством блюд, но и уровнем цифрового сервиса. Клиент ожидает найти актуальное меню, акции, контакты и возможность забронировать стол через интернет без телефонного звонка.

Ресторан «Cappuccino» (г. Астана) — заведение с двумя локациями, итальянской линейкой блюд, кофейной картой и развитым присутствием в социальных сетях. Для повышения качества обслуживания и автоматизации приёма заявок на бронирование требуется современный веб-сайт с серверной частью и панелью администратора.

**Актуальность** темы обусловлена необходимостью автоматизации процессов приёма и обработки бронирований, централизованного хранения данных о меню и акциях, а также предоставления единой точки доступа к информации о заведении.

**Цель дипломного проекта** — разработать веб-систему ресторана «Cappuccino», включающую клиентский сайт, REST API, базу данных SQLite и панель администратора.

**Задачи проекта:**

1. Провести анализ предметной области и существующих решений.
2. Сформулировать функциональные и нефункциональные требования.
3. Спроектировать архитектуру приложения и схему базы данных.
4. Разработать клиентскую часть на React с адаптивным интерфейсom.
5. Разработать серверную часть на Node.js/Express с REST API.
6. Реализовать систему бронирования столов с кодом подтверждения.
7. Реализовать панель администратора для управления бронями.
8. Провести тестирование и подготовить документацию.

**Объект исследования** — процесс информационного обслуживания клиентов ресторана.

**Предмет исследования** — методы и средства разработки веб-приложений с функцией онлайн-бронирования.

**Практическая значимость** — созданное приложение может быть использовано рестораном «Cappuccino» для приёма броней, отображения меню и акций, а также как основа для дальнейшего развития (онлайн-заказ, программа лояльности).

**Структура работы.** Пояснительная записка состоит из введения, пяти глав, заключения, списка источников и приложений.

## 1. АНАЛИЗ ПРЕДМЕТНОЙ ОБЛАСТИ И ПОСТАНОВКА ЗАДАЧИ

### 1.1. Характеристика деятельности ресторана

Ресторан «Cappuccino» работает в сегменте casual dining / coffee shop. Основные направления деятельности:

- приготовление и реализация блюд и напитков;
- обслуживание гостей в зале;
- приём бронирований по телефону и через интернет;
- проведение акций и специальных предложений.

Заведение имеет две локации в Астане:

1. пр. Кабанбай Батыр, 21 (БЦ «Hilton»);
2. пр. Аль-Фараби, 5/1 (БЦ «Nurly Tau»).

Часы работы: понедельник–пятница 09:00–23:00, суббота–воскресенье 10:00–00:00.

### 1.2. Анализ существующих решений

На рынке представлены следующие типы решений для ресторанного бизнеса:

| Решение | Преимущества | Недостатки |
|---------|--------------|------------|
| Конструкторы сайтов (Tilda, Wix) | Быстрый запуск | Ограниченная интеграция с БД и API |
| Агрегаторы (2GIS, Яндекс Еда) | Охват аудитории | Зависимость от платформы, комиссии |
| CRM для HoReCa | Учёт броней | Высокая стоимость, избыточность для малого формата |
| Самописное веб-приложение | Полный контроль, гибкость | Требует разработки |

Для ресторана «Cappuccino» оптимальным является **самописное веб-приложение** с открытым исходным кодом, адаптированное под фирменный стиль и реальные данные заведения.

### 1.3. Анализ пользователей

Выделены две роли:

**Гость (клиент):**
- просматривает информацию о ресторане;
- изучает меню и акции;
- бронирует стол;
- проверяет статус брони по коду.

**Администратор:**
- авторизуется в панели управления;
- просматривает статистику и список броней;
- подтверждает, отменяет или завершает брони.

### 1.4. Функциональные требования

**Для гостя (FR):**

| ID | Требование |
|----|------------|
| FR-01 | Просмотр главной страницы с информацией о ресторане |
| FR-02 | Просмотр меню по категориям с ценами в тенге |
| FR-03 | Просмотр акций |
| FR-04 | Просмотр галереи и контактов |
| FR-05 | Онлайн-бронирование стола |

**Для администратора (AR):**

| ID | Требование |
|----|------------|
| AR-01 | Авторизация по email и паролю |
| AR-02 | Просмотр статистики бронирований |
| AR-03 | Просмотр и фильтрация списка броней |
| AR-04 | Изменение статуса брони |

### 1.5. Нефункциональные требования

| ID | Требование |
|----|------------|
| NFR-01 | Адаптивный интерфейс (desktop, tablet, mobile) |
| NFR-02 | Время отклика API < 500 мс на локальном сервере |
| NFR-03 | Валидация всех входных данных |
| NFR-04 | Защита админ-раздела JWT-токеном |

| NFR-06 | Кроссбраузерность (Chrome, Firefox, Edge) |

### 1.6. Выбор средств разработки

**Frontend:** React 18 — компонентный подход, большое сообщество, TypeScript для типобезопасности, Vite для быстрой сборки, Tailwind CSS для стилизации.

**Backend:** Node.js + Express — единый язык (JavaScript/TypeScript) на клиенте и сервере, простота развёртывания.

**БД:** SQLite — файловая СУБД, не требует отдельного сервера, подходит для малого и среднего трафика.

**Безопасность:** jsonwebtoken, bcryptjs, helmet, express-rate-limit, zod.

## 2. ПРОЕКТИРОВАНИЕ СИСТЕМЫ

### 2.1. Архитектура приложения

Применена **клиент-серверная архитектура** с REST API:

```
[Браузер] → [React SPA] → [Express REST API] → [SQLite]
```

Клиентская часть (Single Page Application) загружается один раз; данные меню, акций и бронирования получаются через HTTP-запросы к API.

В режиме разработки Vite проксирует запросы `/api` на сервер `:3001`.  
В production Express раздаёт статические файлы из `dist/` и обрабатывает API на одном порту.

Подробная схема: `diploma/03_Schemes/architecture.md`.

### 2.2. Проектирование базы данных

Спроектировано 6 таблиц:

1. **locations** — локации ресторана;
2. **menu_categories** — категории меню;
3. **menu_items** — позиции меню;
4. **reservations** — бронирования;
5. **promotions** — акции;
6. **admin_users** — администраторы.

Связи:
- locations (1) → (N) reservations;
- menu_categories (1) → (N) menu_items.

ER-диаграмма: `diploma/03_Schemes/er-diagram.md`.

### 2.3. Проектирование REST API

Спроектировано 11 endpoints (см. `diploma/05_DB_API/api-spec.md`):

**Публичные:** health, menu, locations, promotions, info, availability, reservations (POST), lookup.

**Административные:** login, stats, reservations (GET/PATCH).

### 2.4. Проектирование интерфейса

Дизайн выполнен по макету Figma в тёмной «кофейной» палитре:

- фон: `#1e0f06`;
- акцент: `#c9a84c` (золото);
- шрифты: Cinzel, Playfair Display, Lato.

Структура страницы: Hero → О нас → Меню → Акции → Галерея → Резервация → Контакты → Footer.

Админ-панель: `/admin` — форма входа, дашборд, таблица броней.

### 2.5. Диаграмма бизнес-процесса бронирования

1. Гость заполняет форму (локация, имя, телефон, дата, время, гости).
2. Сервер валидирует данные, проверяет доступность слота.
3. Создаётся запись в БД с кодом CP-XXXXXX.
4. Гость сохраняет код; администратор звонит для подтверждения.
5. Администратор меняет статус в панели на «Подтверждена».

Диаграмма последовательности: `diploma/03_Schemes/booking-flow.md`.

## 3. РЕАЛИЗАЦИЯ ПРОГРАММНОГО ПРОДУКТА

### 3.1. Структура проекта

```
cappuccino-website/
├── src/                    # React frontend
│   ├── App.tsx             # Главная страница
│   ├── pages/AdminPage.tsx # Админ-панель
│   └── api/client.ts       # HTTP-клиент
├── server/src/             # Express backend
│   ├── routes/api.ts
│   ├── services/
│   ├── db/
│   └── validators/
├── server/tests/           # Автотесты
├── data/                   # SQLite (создаётся автоматически)
└── diploma/                # Документация
```

### 3.2. Реализация серверной части

**Модуль database.ts** — инициализация SQLite, создание таблиц, seed-данные (меню, локации, акции, администратор).

**Модуль reservationService.ts** — бизнес-логика:
- проверка часов работы;
- проверка, что дата/время в будущем;
- расчёт занятости слота (max 20 гостей на слот);
- генерация кода CP-XXXXXX;
- CRUD операции для админки.

**Модуль authService.ts** — bcrypt-хеширование, JWT (срок 12 часов).

**Middleware:**
- validateBody / validateQuery (Zod);
- requireAdmin (JWT);
- errorHandler (единый формат ошибок).

**Безопасность:**
- Helmet — HTTP-заголовки;
- CORS — ограничение origin;
- Rate limit: 200 req/15 мин на API, 10 POST бронирований/час.

### 3.3. Реализация клиентской части

**App.tsx** — одностраничный лендинг с секциями. Данные меню, локаций и акций загружаются из API при монтировании; при ошибке используются локальные fallback-данные.

**Форма бронирования** отправляет POST `/api/reservations`. При успехе отображается код брони.

**Проверка брони** — форма ввода кода CP-XXXXXX, запрос GET `/api/reservations/lookup/:code`.

**AdminPage.tsx** — React Router, localStorage для JWT, таблица броней, фильтры по статусу, смена статуса через PATCH.

### 3.4. Ключевые фрагменты кода

**Создание брони (сервер):**
- валидация Zod-схемой createReservationSchema;
- проверка вместимости слота;
- INSERT в таблицу reservations;
- возврат объекта с confirmationCode.

**Авторизация администратора:**
- POST /admin/login → проверка email/password → JWT token;
- все /admin/* routes требуют заголовок Authorization: Bearer.

### 3.5. Развёртывание

```bash
npm install
cp .env.example .env
npm run dev      # development
npm run build && npm start   # production
```

Переменные окружения (.env): PORT, DATABASE_PATH, JWT_SECRET, ADMIN_EMAIL, ADMIN_PASSWORD, CORS_ORIGIN.
